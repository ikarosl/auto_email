# 前端工作台文字 UI 设计

本文档用于描述 AI 邮件询盘系统的前端工作台页面结构。当前阶段先以文字 UI 表达信息架构，不追求视觉细节，后续再落到 Vue 页面和组件。

## 1. 总体布局

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 顶部栏                                                                       │
│ 邮件询盘工作台                         [同步邮件] [刷新] [系统状态: 正常]      │
├───────────────┬──────────────────────────────────────────────────────────────┤
│ 左侧导航       │ 主内容区                                                     │
│               │                                                              │
│ ▸ 工作台概览   │ 当前路由页面内容                                               │
│ ▸ 邮件线程     │                                                              │
│ ▸ 客户管理     │                                                              │
│ ▸ 上下文管理   │                                                              │
│ ▸ AI 回复草稿  │                                                              │
│ ▸ 系统日志     │                                                              │
└───────────────┴──────────────────────────────────────────────────────────────┘
```

### 导航说明

| 模块 | 路由建议 | 主要用途 |
|---|---|---|
| 工作台概览 | `/` | 查看系统健康、询盘数量、待处理任务 |
| 邮件线程 | `/threads` | 按询盘查看客户和我方邮件往来 |
| 客户管理 | `/customers` | 管理客户资产，识别无效客户和有效客户 |
| 上下文管理 | `/contexts` | 查看 AI 每次收到的上下文快照 |
| AI 回复草稿 | `/drafts` | 查看 AI 生成或待生成的回复草稿 |
| 系统日志 | `/logs` | 查看同步、AI 调用、解析诊断等开发日志 |

```text

│ 操作                                                                         │
```

| 操作 | 用途 |

| 生成回复草稿 | 跳转到 AI 回复草稿模块 |

## 4. 客户管理模块

客户管理模块用于维护客户资产的纯净性。由于当前系统采用“邮件先进库，后 AI 判定”的折中方案，所以客户需要有 `unknown / active / invalid` 状态。

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 客户管理                                                    [新增备注] [刷新] │
├──────────────────────────────────────────────────────────────────────────────┤
│ 搜索: [邮箱 / 公司 / 联系人]  状态: [全部 v]  来源: [全部 v]                  │
├──────────────────────────────────────────────────────────────────────────────┤
│ 客户列表                                                                     │
│                                                                              │
│ 邮箱                 名称/公司       状态      询盘数  无效原因        更新时间 │
│ dattilo@gmail.com    Reynaldo        active    2       -               10:30   │
│ ads@example.com      SEO Vendor      invalid   1       广告推广         昨天    │
│ unknown@x.com        -               unknown   1       待判断           昨天    │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 客户详情

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ 客户详情                                                                     │
├──────────────────────────────┬───────────────────────────────────────────────┤
│ 基本信息                      │ 客户询盘                                      │
│ 邮箱: dattilo@gmail.com       │ RF isolator inquiry       ready_for_quote     │
│ 姓名: Reynaldo Dattilo        │ Need delivery confirmation closed             │
│ 公司: -                       │                                               │
│ 状态: active                  │                                               │
│ 无效原因: -                   │                                               │
│ 备注: [可编辑文本]            │                                               │
├──────────────────────────────┴───────────────────────────────────────────────┤
│ 状态历史                                                                     │
│ 10:13 unknown -> active  来源: AI 判定 valid_inquiry                          │
│ 10:29 active -> active   来源: 客户补充有效参数                               │
└──────────────────────────────────────────────────────────────────────────────┘
```

### 客户状态规则

| 状态 | 含义 | 典型来源 |
|---|---|---|
| unknown | 已收到邮件，但还不能确认是否有效客户 | 首次入库、低置信度 AI 判断 |
| active | 有真实询盘或后续被人工确认为有效 | AI `valid_inquiry`、人工确认 |
| invalid | 广告、无关产品、垃圾邮件、商业推广 | AI 高置信度 invalid/commercial/unrelated_product |

## 9. shadcn 使用说明

shadcn 在前端生态里不是传统意义上的“安装一个 UI 库然后直接用”。更准确地说，它是一套组件代码生成/复制方案。

```text
Element Plus:
  安装依赖 -> 从库里 import 组件 -> 组件样式和行为主要由库维护

shadcn / shadcn-vue:
  安装 CLI 或参考组件源码 -> 把组件源码复制到项目里 -> 项目自己拥有和维护组件
```

这意味着：

| 特点 | 说明 |
|---|---|
| 组件归项目所有 | 组件源码在 `src/components/ui` 内，可直接修改 |
| 样式依赖 Tailwind | 通过 class 组合表达视觉 |
| 适合定制 | 后续可以逐步补 Table、Dialog、Tabs、Drawer |
| 不强绑定运行时库 | 不像 Element Plus 那样需要全量组件体系 |

当前项目已经采用这个方向：

```text
apps/admin-web/src/components/ui/
  Button.vue
  Card.vue
  Badge.vue
  Progress.vue
```

后续如果需要更完整的组件，可以引入 `shadcn-vue` CLI，再逐个添加：

```text
button
badge
card
table
tabs
dialog
drawer
textarea
select
```

## 10. 页面实施优先级

| 优先级 | 页面 | 原因 |
|---|---|---|
| P0 | 邮件线程 | 最直接验证邮件归并、方向识别、询盘状态 |
| P0 | 上下文管理 | 最直接验证 AI 判断是否吃到正确上下文 |
| P1 | 客户管理 | 用于清理广告客户、无效客户和真实客户 |
| P1 | AI 回复草稿 | 回复能力上线前的人工审核入口 |
| P2 | 系统日志 | 开发期非常重要，正式期可弱化 |

推荐下一步先做：

```text
1. 邮件线程列表 + 线程详情
2. 上下文快照详情
3. 客户列表和客户状态展示
4. AI 草稿列表
```
