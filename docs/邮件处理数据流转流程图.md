# 邮件处理数据流转流程图

本文档描述系统从开启 IMAP 轮询，到检测新邮件入库、询盘匹配、构建 AI 上下文、调用 AI 并保存建议的完整链路。

## 1. 总览流程图

```text
NestJS 后端启动
  │
  ▼
ImapPollService.onApplicationBootstrap()
  │ 读取 IMAP 配置，解析 mailboxAccount，连接 IMAP
  │
  ▼
ImapPollService.initialSync()
  │ 根据 IMAP_POLL_BOOTSTRAP_MODE 处理历史邮件
  │
  ├─ mark_existing_seen
  │    └─ 只记录当前最大 UID，不处理历史邮件
  │
  └─ process_existing
       └─ 扫描历史 UID，逐封进入 processSingleEmail()

定时轮询开始
  │
  ▼
ImapPollService.startPollingLoop()
  │ 每隔 IMAP_POLL_INTERVAL_MS 调用 pollNewEmails()
  │
  ▼
ImapPollService.pollNewEmails()
  │ 查询邮箱 uidNext，对比 mailbox_sync_states.last_seen_uid
  │ 找到新 UID 列表
  │
  ▼
ImapPollService.processSingleEmail(uid)
  │ 构造 processed email identity
  │ 拉取原始邮件
  │
  ▼
ImapPollService.fetchEmailFromImap()
  │ IMAP fetchOne(source/envelope/internalDate)
  │ mailparser.simpleParser 解析正文、HTML、附件
  │ 生成 InboundEmail
  │
  ▼
PollEmailInboxUseCase.processCandidate()
  │ 检查 processed_emails 幂等
  │
  ▼
ReceiveInboundEmailUseCase.execute()
  │ 去重 messageId
  │ ┌─ EmailContentSanitizer.sanitize() ──────────────┐
  │ │ 1. HTML→纯文本（如有表格则优先用 HTML）          │
  │ │ 2. 引用历史评分移除（阈值 ≥ 70 分）              │
  │ │    • 100分: --- Original Message ---             │
  │ │    • 90分:  On ... wrote: / 在 ... 写道          │
  │ │    • 70分:  连续 3 行 > 引用标记                 │
  │ │    • 0-140分: 邮件头特征评分(From/Sent/To/Subject)│
  │ │ 3. 签名/免责声明行移除（-- / Regards / 本邮件）   │
  │ └──────────────────────────────────────────────────┘
  │ 识别中转邮件真实客户(对@tatasofte.com这样的网站中转提交做固定处理,提取正文中邮箱域名信息)
  │ 解析/创建 EmailThread
  │ 保存 EmailMessage
  │ 保存附件
  │ 匹配或创建 InquiryCase
  │ 关联 inquiry_messages
  │
  ▼
PollEmailInboxUseCase.processCandidate()
  │ 如果无 inquiryCase：跳过 AI
  │ 如果 outbound：只作为上下文入库，跳过 AI
  │ 如果 inbound + inquiryCase：继续 AI 分析
  │
  ▼
PollEmailInboxUseCase.listInquiryEmailMessages()
  │ 查询该询盘下全部关联邮件
  │ 按 receivedAt 升序作为上下文候选
  │
  ▼
AnalyzeEmailWithAiUseCase.execute()
  │ 构造系统提示词
  │ 调用 BuildAiContextUseCase
  │
  ▼
BuildAiContextUseCase.execute()
  │ 查询已有 threadSummary
  │ 选择 recentThreadMessages
  │ 查询 RAG references
  │ 构建结构化 contextPayload
  │ 执行 token budget
  │ 超预算则生成/更新滚动摘要
  │ 保存 AiContextSnapshot
  │ 返回 messages + contextPayload + contextSnapshotId
  │
  ▼
AnalyzeEmailWithAiUseCase.execute()
  │ 调用 EmailAiAnalysisAdapter.analyze(messages)
  │ 解析 AI 原始输出 JSON
  │ Zod schema 运行时校验
  │ 失败时最多重试 2 次
  │ 返回 AI 分析结果
  │
  ▼
DeepseekEmailAnalysisAdapter.analyze()
  │ 通过 OpenAI SDK 兼容接口请求 DeepSeek
  │ 返回模型原始文本
  │
  ▼
PollEmailInboxUseCase.processCandidate()
  │ 保存 ai_decisions
  │ 根据 AI 分析更新 customer.status
  │ 生成/更新 businessSubject
  │ 高置信无效询盘可自动 invalid
  │ 标记 processed_emails 已处理
  │
  ▼
ImapPollService.processSingleEmail()
  │ 输出运行日志：
  │ 邮件入库、询盘 ID、AI classification、suggestedStatus、confidence、contextSnapshotId
```

## 4. 数据表流转

```text
mailbox_accounts
  │ 系统邮箱账户
  ▼
mailbox_sync_states
  │ 记录 last_seen_uid / last_processed_uid
  ▼
processed_emails
  │ UID 幂等处理记录
  ▼
email_threads
  │ 通信层邮件线程
  ▼
email_messages
  │ 单封邮件正文、方向、收发件人、附件数量
  ▼
email_attachments
  │ 附件元数据、存储路径、解析文本、OCR 预留结果
  ▼
customers / organizations
  │ 客户联系人与组织域名
  ▼
inquiry_cases
  │ 业务询盘机会
  ▼
inquiry_messages
  │ 邮件和询盘的关联关系
  ▼
inquiry_context_summaries
  │ 超预算历史邮件滚动摘要
  ▼
ai_context_snapshots
  │ 发给 AI 的最终上下文快照
  ▼
ai_decisions
  │ AI classification / suggestedStatus / reason / extractedRequirements
  ▼
inquiry_status_logs
  │ 实际状态流转日志，仅状态机执行后写入
```

## 5. 当前重要边界

```text
1. AI 的 suggestedStatus 是建议，不等于一定执行状态流转。
2. inbound 邮件会触发 AI 分析；outbound / 内部邮件只作为上下文入库。
3. 无法匹配询盘的我方邮件不会创建新询盘，也不会触发 AI 分析。
4. 新客户入站邮件如果未匹配旧询盘，当前仍会先创建 customer + inquiry_case，再由 AI 判定有效性。
5. PDF 附件优先本地解析；解析为空或失败时，可通过 ATTACHMENT_AI_READER_ENABLED 启用 OpenAI PDF reader。
6. 当前邮件附件可进入上下文全文；历史邮件附件只进入 preview，避免上下文过重。
7. sourceReferences 会记录 email / attachment / summary 等来源，用于追踪 AI 看到了哪些材料。
```
