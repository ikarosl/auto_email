# 询盘归并、人工校正与业务主题整改计划

## 1. 背景问题

当前日志暴露了两个紧急上下文问题：

- 同一家公司多个联系人可能被拆成多个客户和多个询盘，例如 `dykim@rfhic.com` 与 `hshan@rfhic.com`。
- 邮件原始标题直接进入 `inquiryState.subject`，网站表单类标题会误导 AI，例如“你有一个网站表单提交的新询盘”。

这会导致 AI 拿到错误上下文，也会让前端工作台里的询盘线程和客户资产不够可信。

## 2. 总体目标

系统从“自动猜测归并”升级为：

```text
自动初判 + 组织归并 + 人工校正 + 可审计
```

核心能力：

- 企业邮箱域名优先视为同一组织下的多个联系人。
- 近期询盘按 2 个月窗口做自动归并判断。
- 支持手动创建询盘。
- 支持手动修改询盘状态。
- 支持手动关联、移动邮件到正确询盘。
- 支持手动补录真实邮件，并与 IMAP 同步邮件区分。
- 支持 AI 生成业务主题，并允许人工编辑和锁定。

## 3. 数据模型整改

### 3.1 新增组织模型

新增 `organizations` 表，用于表达公司/组织维度。

建议字段：

```text
id
name
domain
status
source
remark
created_at
updated_at
deleted_at
```

规则：

- `rfhic.com` 这类企业邮箱域名可创建组织。
- 公共邮箱域名不作为自动组织归并键，例如 `gmail.com`、`qq.com`、`outlook.com`、`hotmail.com`、`yahoo.com`、`163.com`、`126.com`、`icloud.com`。
- 公共邮箱联系人仍然可以创建客户和询盘，不能被直接屏蔽。
- 如果邮件正文、签名、历史记录或人工判断能确认公共邮箱联系人所属公司，可将该联系人手动绑定到 organization。
- AI 可以提供公共邮箱到组织的归属建议，但不能自动完成强绑定。

### 3.2 Customer 调整为联系人

`customers` 不再同时承担公司和联系人两个含义，而是作为联系人。

建议字段：

```text
organization_id
email
name
domain
company_name
status
invalid_reason
status_updated_at
remark
```

关系：

```text
organization
  └── customers
        └── inquiry_cases
```

### 3.3 InquiryCase 关联组织和主题字段

`inquiry_cases` 建议新增：

```text
organization_id
primary_customer_id
raw_subject
business_subject
business_subject_source
business_subject_locked
business_subject_updated_at
```

字段含义：

| 字段 | 含义 |
|---|---|
| `raw_subject` | 首封或来源邮件的原始标题，只作为证据 |
| `business_subject` | 业务主题，用于 AI 上下文和前端主展示 |
| `business_subject_source` | `raw_email / ai_generated / human` |
| `business_subject_locked` | 人工锁定后 AI 不再覆盖 |
| `organization_id` | 所属组织 |
| `primary_customer_id` | 当前主联系人 |

### 3.4 InquiryMessage 增加审计字段

邮件与询盘的关联需要支持人工移动和审计。

建议新增：

```text
created_by_type
created_by
updated_at
relation_reason
```

`relationType` 继续保留，用于表达：

```text
first_message
reply
manual_link
manual_import
```

### 3.5 手动补录邮件

`EmailSource` 新增：

```text
manual
```

规则：

- 手动补录邮件写入 `email_messages`。
- `source = manual`。
- 不写入 `processed_emails`。
- 不影响 IMAP 的 UID / uidValidity / lastSeenUid / lastProcessedUid。
- 手动补录邮件必须显式选择目标询盘。
- 手动补录邮件可以进入 AI 上下文时间线。

## 4. 自动归并规则

匹配顺序：

```text
1. Message-ID / In-Reply-To / References / emailThreadId 精确线程匹配
2. 已有关联邮件线程匹配
3. 同一联系人邮箱近期打开询盘
4. 同一企业组织域名近期打开询盘
5. 多个候选时进入人工确认
6. 无候选时创建新询盘
```

近期窗口：

```text
2 个月
```

同组织归并规则：

- 同企业组织 2 个月内只有一个打开询盘：自动归并。
- 同企业组织 2 个月内存在多个打开询盘：不自动归并，进入人工确认。
- 公共邮箱域名不作为自动组织归并键，但公共邮箱联系人仍按联系人邮箱自身处理。
- 公共邮箱联系人如果后续被人工绑定到组织，才可通过组织视角参与人工筛选和展示。

## 5. 人工校正能力

### 5.1 手动创建询盘

前端需要提供手动创建询盘入口。

必填：

```text
organization 或 customer
businessSubject
status
```

可选：

```text
rawSubject
productType
remark
```

### 5.2 修改询盘状态

沿用当前状态机。

前端调用：

```text
POST /api/inquiries/:id/transitions
```

需要记录：

```text
changedByType = human
changedBy
reason
```

### 5.3 手动关联邮件

支持把已有邮件关联到询盘。

建议接口：

```text
POST /api/inquiries/:id/messages
```

模式：

```text
link_existing_email
create_manual_email
```

### 5.4 移动错误关联邮件

如果某个询盘线程下出现无关邮件，允许人工移动。

规则：

- 不做静默隐藏。
- 不能移出后无归属。
- 移动时必须选择目标询盘，或创建新询盘。

建议接口：

```text
POST /api/inquiry-messages/:id/move
```

请求示例：

```json
{
  "targetInquiryCaseId": "inquiry_xxx",
  "reason": "该邮件属于另一个采购主题",
  "changedBy": "operator@example.com"
}
```

## 6. 业务主题策略

### 6.1 原始标题与业务主题分离

邮件原始标题继续保留：

```text
email_messages.subject
```

询盘业务主题使用：

```text
inquiry_cases.business_subject
```

AI 上下文中的：

```text
inquiryState.subject
```

应使用：

```text
businessSubject ?? cleanedRawSubject
```

### 6.2 AI 生成业务主题

AI 可以根据邮件正文和上下文生成业务主题。

规则：

- AI 生成后写入 `business_subject`。
- `business_subject_source = ai_generated`。
- `business_subject_locked = false`。
- 人工可以编辑并锁定。
- 锁定后 AI 不再覆盖。

### 6.3 recentThreadMessages 的 subject

`recentThreadMessages[].subject` 保留原始邮件标题。

原因：

- 它是证据字段。
- 不应被 AI 改写。
- 如果后续需要展示业务含义，应新增独立字段，不覆盖原始 subject。

## 7. API 规划

### 7.1 Organization

```text
GET /api/organizations

GET /api/organizations/:id/customers
GET /api/organizations/:id/inquiries
```

### 7.2 Customer

```text
GET /api/customers
GET /api/customers/:id
PATCH /api/customers/:id
```

新增能力：

- 修改联系人所属组织。
- 修改联系人备注。
- 查看联系人关联询盘。

### 7.3 Inquiry

```text
POST /api/inquiries
PATCH /api/inquiries/:id
POST /api/inquiries/:id/transitions
POST /api/inquiries/:id/messages
```

### 7.4 InquiryMessage

```text
POST /api/inquiry-messages/:id/move
```

### 7.5 EmailMessage

```text
POST /api/email-messages/manual
```

或复用：

```text
POST /api/inquiries/:id/messages
```

通过 `mode = create_manual_email` 区分。

## 9. 测试场景

### 9.1 同组织归并

场景：

```text
dykim@rfhic.com 首次询盘
hshan@rfhic.com 后续参与同一主题
```

预期：

- 两个联系人属于同一 organization。
- 如果 2 个月内只有一个打开询盘，自动归并到该询盘。

### 9.2 公共邮箱真实客户

场景：

```text
buyer@gmail.com 首次询盘
```

预期：

- 创建 customer 和 inquiryCase。
- 不因为 gmail.com 是公共邮箱而屏蔽。
- 不通过 gmail.com 域名自动归并到其他 gmail 联系人。
- 可人工绑定到某个 organization。

### 9.3 多询盘冲突

场景：

```text
rfhic.com 在 2 个月内有两个打开询盘
新邮件来自第三个联系人
```

预期：

- 不自动归并。
- 标记人工确认。

### 9.4 网站表单标题

场景：

```text
subject = 你有一个网站表单提交的新询盘
body 中包含真实需求
```

预期：

- `raw_subject` 保存原始标题。
- AI 生成 `business_subject`。
- `inquiryState.subject` 使用业务主题。

### 9.5 主题锁定

场景：

```text
人工编辑并锁定 business_subject
后续新邮件进入
```

预期：

- AI 不覆盖已锁定主题。

### 9.6 手动补录邮件

场景：

```text
业务人员补录一封真实沟通邮件
```

预期：

- `email_messages.source = manual`。
- 不写入 `processed_emails`。
- 不影响 IMAP 同步状态。
- 可进入目标询盘上下文。

### 9.7 移动错误关联邮件

场景：

```text
询盘 A 中出现应属于询盘 B 的邮件
```

预期：

- 人工移动到询盘 B。
- 询盘 A 后续上下文不再包含该邮件。
- 移动操作有审计记录。

## 10. 实施顺序

建议分阶段实现：

```text
1. 数据模型迁移：Organization、业务主题字段、手动邮件 source、关联审计字段
2. 自动归并规则：组织域名、2 个月窗口、多候选人工确认
3. 业务主题生成：AI 生成、自动写入、人工锁定
4. 后端人工校正接口：手动创建询盘、关联/移动邮件、补录邮件
5. 前端页面：组织视角、询盘详情操作、邮件移动/补录入口
6. 回归测试与历史数据人工校正
```

本阶段不做自动合并历史脏数据，历史拆错的询盘通过人工移动/合并能力逐步修复。
