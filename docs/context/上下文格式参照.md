# AI 邮件分析上下文格式参照

当前发送给 AI 的 Chat API 入参保持两段结构：

- `system`: 固定规则、身份、状态建议边界、输出要求。
- `user`: 结构化 JSON 字符串。下面为了阅读方便展示为对象，实际发送时会通过 `JSON.stringify(contextPayload, null, 2)` 转为字符串。

内部数据库 ID、token 估算、sourceReferences 等追踪信息不进入 AI prompt，只保存在 debug log 和上下文快照 metadata 中。

## 字段说明

`threadSummary` 是被压缩的早期邮件历史。它只在历史线程超过预算时出现；如果线程很短，payload 可以没有该字段。

`coveredMessageCount` 和 `coveredTimeRange` 用于说明摘要覆盖范围，方便系统判断后续哪些邮件已经进入滚动摘要。

`knownFacts` 保存已确认事实，例如产品、频段、功率、数量、连接器、价格和交期。

`customerDecisions` 保存客户已经明确接受、拒绝、取消或确认的信息。

`ourCommitments` 保存我方已经承诺、报价、发送资料或说明过的信息。

`openQuestions` 保存仍未解决的问题；如果没有未决问题，使用空数组。

`recentThreadMessages` 是近期原文窗口，按 `receivedAt` 从旧到新排序，不包含已经被 `threadSummary` 覆盖且超预算的早期邮件。

`currentEmail` 是本次必须分析的邮件，永远保留，不参与裁剪。

## Token 预算公式

当前上下文预算由 `.env` 控制：

```env
AI_CONTEXT_MAX_TOKENS=64000
AI_CONTEXT_INPUT_TOKEN_RATIO=0.8
AI_CONTEXT_OUTPUT_TOKEN_RATIO=0.1
AI_CONTEXT_SAFETY_TOKEN_RATIO=0.1
```

实际计算公式：

```text
inputTokenBudget = floor(AI_CONTEXT_MAX_TOKENS * AI_CONTEXT_INPUT_TOKEN_RATIO)

fixedTokens =
  systemPrompt tokens
  + JSON.stringify(payload without recentThreadMessages) tokens

recentThreadMessagesBudget = max(0, inputTokenBudget - fixedTokens)
```

`systemPrompt`、`inquiryState`、`threadSummary`、`ragReferences`、`currentEmail`、`outputInstruction` 都属于固定区，实际计算 token，不做粗略比例预留。

`recentThreadMessages` 按时间从新到旧纳入预算，最终发送给 AI 时仍按 `receivedAt` 从旧到新排序。超过预算的早期邮件交给 AI 生成或更新 `threadSummary`。

## 摘要生成配置

滚动摘要由 AI 生成，不使用规则摘要。摘要模型和提示词可通过 `.env` 覆盖：

```env
AI_CONTEXT_SUMMARY_MODEL=deepseek-v4-pro
AI_CONTEXT_SUMMARY_SYSTEM_PROMPT=
```

`AI_CONTEXT_SUMMARY_SYSTEM_PROMPT` 为空时，系统使用内置摘要提示词。该提示词要求摘要只保留业务相关信息，包括产品参数、数量、价格、交期、客户决定、我方承诺、风险和未决问题，不允许生成内部数据库 ID 或 token 信息。
